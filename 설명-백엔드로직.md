# Hermes Connection 백엔드 로직 설명

이 문서는 플러그인이 Hermes 서버와 통신하는 방식, 세션/프로젝트/모델/검색/문맥/승인 로직을 설명합니다.

여기서 말하는 백엔드는 플러그인 내부의 서버 통신 로직입니다. 별도 Node 서버가 플러그인 안에 있는 것은 아닙니다.

## 1. 기본 원칙

플러그인은 컴퓨터 제어를 직접 하지 않습니다.

```text
Obsidian plugin
  -> Hermes server에 요청

Hermes server
  -> 모델 호출
  -> MCP/tool 호출
  -> 파일 접근
  -> 터미널 실행
  -> 승인 처리
```

이 원칙 덕분에 iPad/Android/Windows/macOS에서 같은 구조를 유지할 수 있습니다.

## 2. 설정 저장

플러그인 설정은 Obsidian plugin data에 저장됩니다.

예상 위치:

```text
<Vault>/.obsidian/plugins/hermes-connection/data.json
```

개인용 편의를 위해 다음 값은 `data.json`에 저장될 수 있습니다.

```text
hermesServerUrl
dashboardUsername
dashboardPassword
lastSessionId
lastModelId
reasoningMode
accessMode
visibleModelIds
useLiveWebSocket
```

주의:

- `data.json`은 GitHub에 커밋하지 않습니다.
- iCloud/NAS/WebDAV로 Vault를 동기화하면 iPad/iPhone에도 설정이 넘어갑니다.
- 여러 사람이 쓰는 Vault라면 password 저장 방식은 바꾸는 것이 좋습니다.

## 3. Server URL

플러그인은 로컬 자동 감지를 기본 연결 방식으로 사용하지 않습니다.

이유:

```text
Mac에서 127.0.0.1 -> Mac 자신
iPad에서 127.0.0.1 -> iPad 자신
```

따라서 원격 사용에서는 사용자가 명시적으로 서버 URL을 입력합니다.

예시:

```text
http://192.168.0.10:9119
http://100.123.26.117:9119
```

모델/세션/메시지/승인 endpoint 설정값은 내부 기본값으로 유지하지만 일반 설정 화면에는 노출하지 않습니다. 사용자가 직접 만질 경우 연결 실패 원인이 되기 쉽기 때문입니다.

## 4. Dashboard auth

원격 dashboard 모드에서는 username/password로 로그인합니다.

흐름:

```text
GET  /api/auth/providers
POST /auth/password-login
POST /api/auth/ws-ticket
WS   /api/ws?ticket=...
```

### provider 선택

플러그인은 `/api/auth/providers`에서 password login provider를 찾습니다.

우선순위:

1. 설정된 `dashboardAuthProvider`
2. `supports_password`가 true인 provider
3. 첫 번째 provider

보통 username/password 방식이면 `basic` provider가 선택됩니다.

### requestUrl 사용

Obsidian 환경에서는 일반 `fetch`가 CORS/cookie 문제를 만날 수 있습니다.

그래서 플러그인은 가능하면 Obsidian `requestUrl` fallback을 사용합니다.

장점:

- 모바일에서 CORS 문제 감소
- dashboard cookie 처리 가능
- Hermes server URL이 private network/Tailscale이어도 접근 가능

## 5. Live WebSocket

내부 설정 이름은 `useLiveWebSocket`입니다. 예전 배포본의 `useDesktopGateway` 값이 `data.json`에 남아 있으면 로드 시 `useLiveWebSocket`으로 자동 마이그레이션합니다.

플러그인은 dashboard에서 받은 ticket으로 `/api/ws`에 연결합니다.

```text
ws://server:9119/api/ws?ticket=...
```

이 기능은 `hermes serve`가 제공하는 실시간 이벤트 endpoint를 사용하는 것입니다. 터미널 명령어 `hermes gateway`로 관리하는 메시징 gateway와는 별개입니다.

일반 설정 화면에는 별도 토글을 두지 않습니다. 기본값은 사용이며, 연결 실패 시 REST/SSE 방식으로 자동 폴백합니다.

주요 RPC:

```text
session.create
session.resume
prompt.submit
approval.respond
projects.tree
projects.project_sessions
session.delete
```

## 6. 세션 생성과 재개

### 새 세션

새 채팅 또는 첫 메시지 전송 시:

```text
session.create
```

응답에서 runtime session id와 stored session id를 받을 수 있습니다. 플러그인은 둘 사이를 `liveRuntimeSessions` map에 저장합니다.

### 기존 세션

기존 세션에 메시지를 보내기 전:

```text
session.resume
```

resume이 실패하면 기존 id를 그대로 사용합니다.

### 메시지 전송

```text
prompt.submit
```

보내는 텍스트는 대략 다음 구조입니다.

```text
[Obsidian instructions and context]
...

[User message]
...
```

단 UI에는 이 전체 prompt를 보여주지 않고, 사용자 원문만 보여줍니다.

## 7. 이벤트 처리

Hermes live WebSocket 이벤트를 UI line으로 변환합니다.

주요 매핑:

```text
message.delta       -> assistant text
thinking.delta      -> reasoning
reasoning.delta     -> reasoning
subagent.thinking   -> reasoning
tool.start          -> tool
tool.progress       -> tool
tool.complete       -> tool
approval.request    -> approval
status.update       -> system/status
error               -> system/error
```

### reasoning.available 무시

일부 상황에서 최종 답변과 유사한 내용이 `reasoning.available`로 다시 들어올 수 있습니다.

이를 thinking으로 렌더링하면 답변이 reasoning에 덮이거나 중복 표시됩니다.

현재는 `reasoning.available`을 무시합니다.

Hermes 업데이트 후 thinking/answer 분리가 이상하면 이 매핑을 먼저 확인하세요.

## 8. 완료 이벤트와 입력 잠금 해제

플러그인은 다음 이벤트를 terminal event로 봅니다.

```text
message.complete
message.completed
assistant.completed
approval.request
error
turn.complete
run.complete
response.completed
session.idle
agent.idle
task.complete
```

terminal event를 받으면 send lock을 풉니다.

Hermes가 완료 이벤트를 보내지 않고 멈추는 경우를 대비해 timeout도 둡니다.

## 9. 세션 목록

기본 세션 목록:

```text
GET /api/sessions?limit=200
```

응답은 다음 중 하나의 배열로 파싱합니다.

```text
sessions
data
items
result
```

세션 필터:

- message count가 0인 세션은 기본적으로 숨김
- `api_server` source 중 title이 없거나 `Obsidian:`으로 시작하지 않는 세션은 숨김
- Hermes/TUI 세션은 유지

## 10. 프로젝트 그룹

Hermes project tree:

```text
projects.tree
projects.project_sessions
```

현재 확인된 특징:

- Hermes project 목록은 내려올 수 있음
- project_sessions가 항상 세션을 채워주지는 않을 수 있음
- `scoped_session_ids`가 비어 있을 수 있음

fallback:

세션의 `cwd` 또는 `gitRepoRoot`가 project path와 같거나 하위 경로이면 해당 project 세션으로 묶습니다.

```text
session.cwd == project.path
session.cwd startsWith project.path + "/"
session.gitRepoRoot == project.path
```

project에 세션이 0개여도 project select에는 표시합니다.

## 11. 세션 검색

History modal 검색은 server search를 먼저 사용합니다.

```text
GET /api/sessions/search?q=...
```

확인된 응답 형태:

```json
{
  "results": [
    {
      "session_id": "...",
      "snippet": "...",
      "role": "tool",
      "model": "gpt-5.4-mini"
    }
  ]
}
```

플러그인은 `results`도 파싱합니다.

검색 결과에 title이 없으면:

1. session id로 기존 세션 목록과 매칭
2. title/model/message count/cwd 보강
3. 그래도 title이 없으면 session id 표시

server search 결과가 없거나 실패하면 local fallback을 사용합니다.

local fallback:

- 최대 80개 세션만 대상
- 제목/preview 먼저 검색
- 필요하면 `listMessages(sessionId)`로 user/assistant 내용 일부 cache

## 12. 세션 삭제

History modal에서 세션 삭제를 지원합니다.

우선순위:

1. WebSocket RPC `session.delete`
2. REST `DELETE /api/sessions/{sessionId}`

안전장치:

- 현재 active session 삭제 금지
- browser confirm 후 삭제
- 삭제 후 session list refresh
- search cache 제거

주의:

세션 삭제는 실제 Hermes 세션과 메시지를 삭제할 수 있습니다. 테스트할 때 기존 실제 세션을 삭제하지 말고, 테스트용 세션을 만든 뒤 검증해야 합니다.

## 13. 모델 목록

원격 dashboard 모드:

```text
GET /api/model/options
```

응답 provider/model 목록을 flatten합니다.

내부 model id:

```text
provider\0model
```

예:

```text
openai-codex\0gpt-5.5
nous\0model-name
custom:ollama-local\0gemma4:e2b-mlx
```

전송 시:

- provider는 `\0` 앞부분
- model은 `\0` 뒷부분

## 14. visible models

모델 목록이 너무 길면 visible model 설정으로 드롭다운 표시 대상을 줄입니다.

현재 방식:

- Hermes desktop local storage에서 visible model 설정을 읽을 수 있으면 단방향 동기화
- 플러그인 설정에도 visible model list 저장 가능

주의:

Hermes 앱의 model edit toggle과 완전한 양방향 동기화는 Hermes 쪽 공식 API가 있어야 안전합니다.

코어 파일을 직접 수정해서 API를 추가하면 Hermes 업데이트 때 깨질 수 있습니다.

## 15. Obsidian context metadata

사용자 메시지와 함께 Obsidian metadata를 보냅니다.

예시 필드:

```text
vaultName
vaultRoot
activeFile
activeFolder
selectedText
mentionedFiles
attachedFiles
folderFilePaths
folderFileCount
ragRecommended
ragSearchProvider
ragSearchScopeType
ragSearchScopePath
workingScope
reasoningMode
accessMode
```

원칙:

- Vault 내부 경로는 기본적으로 vault-relative path를 보냅니다.
- 파일 내용이 필요한 경우 Obsidian `vault.read(file)`로 읽어서 attachedFiles에 넣습니다.
- 서버가 직접 파일을 읽어야 하는 경우를 위해 absolutePath hint를 일부 제공할 수 있습니다.

## 16. 파일/폴더 문맥 처리

### current-file

현재 active Markdown file 또는 선택 영역을 읽습니다. 별도 `Attach active note` 토글은 두지 않고, context scope가 `current-file`이면 현재 노트 본문/선택 영역을 첨부합니다.

### mentioned file

`@[[note]]` 또는 mention UI로 고른 파일을 읽습니다.

### current-folder / vault-folder

폴더 안 Markdown 파일 목록을 수집합니다.

작은 범위:

- 일부 파일 내용을 inline context로 첨부

큰 범위:

- 전체 내용을 붙이지 않음
- file path 목록과 RAG/search hint 제공

### vault-all

Vault 전체 내용을 프롬프트에 붙이지 않습니다.

대신:

```text
ragRecommended: true
ragSearchScopeType: vault-all
ragSearchScopePath: ""
```

이렇게 Hermes가 검색 도구를 활용할 수 있게 힌트를 줍니다.

## 17. docsearch-mcp / RAG

플러그인이 docsearch-mcp를 직접 실행하지 않습니다.

역할:

```text
Plugin
  -> RAG가 필요해 보이는 범위 metadata 전달

Hermes
  -> MCP/tool 판단
  -> docsearch-mcp query
  -> 검색 결과를 답변에 활용
```

큰 폴더나 Vault 전체 질문에서 플러그인이 할 일:

- file list 제공
- scope metadata 제공
- ragRecommended 제공
- search provider hint 제공

플러그인에서 모든 검색을 직접 수행하지 않는 이유:

- Hermes가 tool orchestration을 맡는 것이 구조적으로 자연스럽습니다.
- 작은 로컬 모델/큰 모델에 따라 tool 판단 능력이 달라질 수 있습니다.
- 플러그인은 기기 독립 UI 역할을 유지해야 합니다.

## 18. 승인/거절

Hermes event:

```text
approval.request
```

플러그인 처리:

- 채팅 UI에 Approve/Reject 버튼 표시
- 버튼 상태 관리
- WebSocket approval이면:

```text
approval.respond
```

REST fallback이면:

```text
POST /api/sessions/{sessionId}/approvals/{approvalId}
```

Ask mode에서는 Hermes approval gate가 실제 변경 작업을 멈추고 사용자 승인을 기다립니다.

## 19. 경로 관련 유지보수

macOS:

```text
/Users/user/...
```

Windows:

```text
C:\Users\user\...
```

Vault 경로:

- 기본은 vault-relative path
- 경로 구분자는 `/`로 normalize
- absolute path는 서버 검증/hint 용도로만 제한적으로 사용

NAS/iCloud/WebDAV:

- Vault 내부 상대 경로가 같으면 플러그인 문맥 전달은 유지됩니다.
- 서버가 직접 파일 시스템을 읽어야 하는 경우 서버 측 mount path가 다를 수 있으므로 주의해야 합니다.

## 20. 유지보수 체크포인트

Hermes 업데이트 후 확인:

- dashboard auth path
- ws-ticket API
- live event type
- session RPC 이름
- projects RPC 응답 구조
- search API 응답 구조
- model options API 응답 구조

Obsidian 업데이트 후 확인:

- `requestUrl`
- `MarkdownRenderer.renderMarkdown`
- `workspace.on("file-open")`
- `workspace.on("active-leaf-change")`
- 모바일 plugin API 제한

## 21. 테스트 절차

정적 검사:

```bash
node --check hermes-connection/main.js
git diff --check
```

서버 API:

```text
GET /api/status
GET /api/sessions?limit=20
GET /api/sessions/search?q=git
POST /api/auth/ws-ticket
WS /api/ws
```

UI:

- 세션 목록 표시
- 프로젝트 select 표시
- History modal 열림
- 검색 결과 표시
- Activity 묶음 표시
- 완료 Activity 애니메이션 없음
- approval 버튼 표시
- session delete confirm 표시

주의:

삭제 테스트는 실제 세션이 아니라 테스트용 세션으로만 수행합니다.
