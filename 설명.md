# Hermes Connection 개발 설명

이 문서는 `Hermes Connection` 플러그인의 작동 원리와 유지보수 포인트를 설명합니다.

대상 독자는 플러그인을 수정하거나, Hermes/Obsidian 업데이트 후 문제가 생겼을 때 원인을 추적해야 하는 개발자입니다.

## 1. 목표

`Hermes Connection`의 목표는 Obsidian을 Hermes 홈서버의 원격 채팅 클라이언트로 사용하는 것입니다.

핵심 설계:

```text
Obsidian plugin
  -> 채팅 UI
  -> 현재 노트/멘션/폴더 문맥 수집
  -> Hermes session API 또는 live gateway로 전송

Hermes server
  -> 모델 호출
  -> 도구 실행
  -> 파일 접근
  -> 터미널 실행
  -> 승인/거절 처리
```

플러그인은 컴퓨터 제어를 직접 구현하지 않습니다.  
`child_process`, 로컬 stdio ACP, Electron 전용 API는 기본 기능에서 제외하는 방향입니다.

이유:

- iPad/iPhone/Android에서는 로컬 프로세스 실행이 불가능합니다.
- Obsidian 모바일은 Node/Electron API에 의존할 수 없습니다.
- 원격 Hermes 서버가 실행 주체가 되어야 기기별 차이를 줄일 수 있습니다.

## 2. 주요 파일

```text
hermes-connection/
  manifest.json
  main.js
  styles.css
```

### manifest.json

Obsidian 플러그인 메타데이터입니다.

중요 필드:

```json
{
  "id": "hermes-connection",
  "name": "Hermes Connection",
  "isDesktopOnly": false
}
```

`isDesktopOnly: false`이므로 모바일 Obsidian도 고려해야 합니다.

### main.js

플러그인의 모든 런타임 로직이 들어 있습니다.

현재 배포본은 번들된 단일 JS 파일입니다. 유지보수를 쉽게 하려면 장기적으로 TypeScript 원본 구조를 복원하거나 새로 구성하는 것이 좋습니다.

주요 책임:

- Obsidian View 등록
- 설정 탭 생성
- Hermes 연결
- 모델 목록 로드
- 세션 목록 로드
- 메시지 전송
- live gateway WebSocket 연결
- streaming event 처리
- mention UI 처리
- Obsidian context 생성
- 승인/거절 UI 처리

### styles.css

채팅 UI, mention menu, thinking block, composer, toolbar 스타일입니다.

최근 UI 방향:

- 상단 버튼은 박스가 아니라 아이콘처럼 보이게 처리
- 입력창은 낮고 하단에 붙게 처리
- 전송 버튼도 배경 없는 아이콘 형태
- iPad/모바일 폭에서 공간 절약

## 3. 전체 동작 흐름

```mermaid
flowchart TD
  A["User types in Obsidian"] --> B["Plugin reads selected context"]
  B --> C["Build system message and metadata"]
  C --> D["Login to Hermes dashboard if needed"]
  D --> E["Request WebSocket ticket"]
  E --> F["Connect to /api/ws"]
  F --> G["session.create or session.resume"]
  G --> H["prompt.submit"]
  H --> I["Hermes streams events"]
  I --> J["Plugin renders message/reasoning/tool/approval"]
```

## 4. Hermes 통신 방식

### 4.0. Server URL과 설정 저장

플러그인은 `127.0.0.1` 기반 로컬 Hermes 자동 감지를 기본 연결 경로로 사용하지 않습니다. 모바일에서 `127.0.0.1`은 Hermes 서버 컴퓨터가 아니라 모바일 기기 자기 자신이기 때문입니다.

현재 기본 원칙:

```text
사용자가 입력한 Hermes server URL
  -> dashboard login
  -> ws-ticket 발급
  -> /api/ws WebSocket 연결
```

예를 들어 iPad와 Mac이 모두 같은 Mac Studio Hermes 서버를 바라보려면 두 기기 모두 다음처럼 Tailscale 주소를 사용합니다.

```text
http://100.123.26.117:9119
```

개인용 편의를 위해 `dashboardUsername`, `dashboardPassword`, `gatewayToken`은 플러그인 `data.json`에 저장합니다. 이 파일은 Vault 동기화 대상이 될 수 있으므로 iCloud/NAS/WebDAV를 통해 iPad/iPhone에도 같은 연결 설정이 넘어갑니다.

### 4.1. Dashboard login

원격 dashboard 모드에서는 username/password를 사용합니다.

흐름:

```text
GET /api/auth/providers
POST /auth/password-login
POST /api/auth/ws-ticket
WebSocket /api/ws?ticket=...
```

플러그인은 Obsidian의 `requestUrl`을 사용해 HTTP 요청을 보냅니다.  
이 방식은 모바일에서 CORS/cookie 문제가 생길 가능성을 줄입니다.

주의:

- Hermes가 auth API 경로를 바꾸면 이 부분을 수정해야 합니다.
- password login provider 이름이 바뀌면 provider 선택 로직을 확인해야 합니다.
- 로그인 실패가 반복되면 Hermes가 429 rate limit을 줄 수 있습니다.

### 4.2. WebSocket live gateway

live gateway는 Hermes의 실시간 이벤트를 받기 위해 사용합니다.

주요 RPC:

```text
session.create
session.resume
prompt.submit
approval.respond
```

주요 이벤트:

```text
message.delta
message.complete
reasoning.delta
reasoning.available
thinking.delta
tool.start
tool.progress
tool.complete
approval.request
status.update
error
```

현재 플러그인 처리:

- `message.delta` -> assistant 답변
- `reasoning.delta`, `thinking.delta` -> thinking/reasoning 블록
- `reasoning.available` -> 무시
- `tool.*` -> 도구 로그
- `approval.request` -> 승인/거절 버튼
- `error` -> 에러 메시지

`reasoning.available`을 무시하는 이유:

일부 Hermes 이벤트에서는 최종 답변이 `message.delta`로 이미 온 뒤, 같은 내용이 `reasoning.available`로 다시 오는 경우가 있었습니다. 이를 thinking으로 렌더링하면 답변이 thinking 블록에 중복 표시됩니다.

Hermes 업데이트 후 답변/thinking 분리가 깨지면 event mapping을 먼저 확인해야 합니다.

## 5. 세션 동작

플러그인은 Hermes 세션을 가져오고 생성합니다.

기능:

- 세션 목록 불러오기
- 새 세션 만들기
- 기존 세션 선택
- 세션 드롭다운 클릭/포커스 시 조용히 동기화
- 주기적 동기화

주의:

- Hermes 앱에서 세션 rename/delete를 하면 플러그인 목록과 동기화되어야 합니다.
- 세션 드롭다운을 열 때마다 최신 목록을 가져오도록 구현되어 있습니다.
- Hermes API 응답 구조가 바뀌면 session parser를 수정해야 합니다.

## 6. 모델 목록 동작

원격 dashboard 모드에서 모델 목록은 Hermes dashboard의 모델 옵션 API를 사용합니다.

현재 방향:

```text
GET /api/model/options
```

그 뒤 provider/model 목록을 flatten해서 Obsidian 드롭다운에 표시합니다.

모델 id 내부 표현은 provider와 model을 함께 담기 위해 구분자를 사용합니다.

예:

```text
openai-codex\0gpt-5.5
nous\0some-model
custom:ollama-local\0gemma4:e2b-mlx
```

유지보수 포인트:

- Hermes 모델 옵션 API 구조가 바뀌면 listModels 로직 수정
- provider slug가 바뀌면 provider label 처리 수정
- visible model API가 공식 제공되면 현재 local storage 읽기 방식은 제거 가능

## 7. Mention 기능

mention 기능은 사용자가 채팅 입력창에서 Obsidian 문맥을 고르는 UI입니다.

현재 지원:

```text
Active Note
Notes
Folders
Vault
No Context
```

Copilot/Agent Client와 비슷한 사용감을 목표로 합니다.

### 7.1. Active Note

현재 보고 있는 Markdown 파일을 context로 선택합니다.

중요 요구:

현재 파일 선택 후 사용자가 보는 파일을 바꾸면, current file context도 따라가야 합니다.

이를 위해 플러그인은 Obsidian workspace 이벤트를 구독합니다.

```text
active-leaf-change
file-open
```

이 이벤트에서 마지막 active Markdown file path를 갱신하고, mention chip도 갱신합니다.

### 7.2. Notes

Vault의 Markdown 파일을 검색합니다.

검색 대상:

- basename
- vault-relative path

선택 시 내부 토큰은 대략 다음 형태입니다.

```text
@[[path/to/note.md|note]]
```

서버에 보낼 때는 파일 내용 일부와 metadata가 함께 구성됩니다.

### 7.3. Folders

폴더 mention은 파일 mention과 다릅니다.

작은 폴더:

- 폴더 내 Markdown 파일 목록 전달
- 일부 파일 내용 inline context로 전달

큰 폴더:

- 모든 내용을 프롬프트에 넣으면 비효율적
- 파일 목록과 metadata를 전달하고, 실제 검색은 Hermes의 RAG/MCP 도구를 쓰도록 유도

현재 구현은 폴더 안 Markdown 파일 목록을 최대 일정 개수까지 표시하고, 내용 첨부는 제한합니다. `current-folder`, `vault-folder`에서는 앞쪽 일부 노트 본문을 미리보기로 붙일 수 있지만, `vault-all`에서는 임의의 앞 12개 파일 본문을 붙이지 않고 파일 목록과 RAG-aware metadata 중심으로 보냅니다.

핵심 필드:

```text
ragRecommended: true
ragSearchProvider: obsidian-docsearch/docsearch-mcp
ragSearchScopeType: current-folder | vault-folder | vault-all
ragSearchScopePath: Vault 상대 경로
folderFilePaths: 검색 범위의 Vault 상대 파일 목록
folderFileCount: 검색 범위의 Markdown 파일 수
```

중요:

- `vault-all`도 RAG-aware 처리 대상입니다.
- `vault-all`은 전체 Vault를 임의 순서로 일부 본문 첨부하지 않고, 검색 범위와 파일 목록을 알려 Hermes/docsearch-mcp가 직접 탐색하도록 유도합니다.

## 8. Context Router 개념

이 플러그인의 context router는 사용자의 질문과 mention 상태를 보고 어떤 문맥을 Hermes에 보낼지 결정하는 계층입니다.

개념적 흐름:

```text
사용자 질문
  -> mention 분석
  -> 현재 파일/현재 폴더/폴더/노트/Vault 판별
  -> 크기 계산
  -> 작으면 inline context
  -> 크면 metadata + RAG hint + Hermes MCP 검색 유도
```

Context Router는 두 종류의 신호를 봅니다.

```text
1. 명시적 mention
   예: @현재파일, @현재폴더, @폴더:..., @전체Vault

2. 자연어 범위 표현
   예: 현재 폴더, 이 폴더, 지금 폴더, current folder, this folder
```

따라서 사용자가 다음처럼 말하면 mention이 없어도 `current-folder` scope로 전환됩니다.

```text
현재 폴더에 좀 오래된 글은 좀 삭제해줘
이 폴더에서 임시 메모를 찾아줘
지금 폴더 구조를 보여줘
```

이때 플러그인은 현재 active Markdown file의 parent folder를 기준으로 `folderFilePaths`, `folderFileCount`, `ragRecommended` metadata를 구성합니다.

현재 inline context가 유리한 경우:

- 현재 파일
- 선택 영역
- 짧은 파일 mention
- 작은 폴더

RAG/MCP가 유리한 경우:

- Vault 전체
- 큰 폴더
- 긴 문서 다수
- PDF/코드/첨부파일 검색
- 최신 인덱스 기반 검색

장기적으로는 다음 구조가 가장 안정적입니다.

```text
Obsidian plugin
  -> active file and selected mentions inline
  -> large scope metadata

Hermes
  -> obsidian-docsearch/docsearch-mcp 같은 MCP RAG search
  -> tool execution
  -> answer generation
```

### 8.1. 현재 구현된 RAG-aware 동작

플러그인은 큰 범위 문맥을 감지하면 system message에 다음 의도를 포함합니다.

```text
attached note snippets are only a preview
if the answer requires content lookup, comparison across many notes, or files not fully attached,
use Hermes MCP/RAG document search before answering
prefer the configured obsidian-docsearch/docsearch-mcp MCP server
```

폴더/Vault 구조 블록도 다음 제목으로 생성됩니다.

```text
[Folder/Vault structure and RAG hint]
```

이 블록에는 다음 정보가 포함됩니다.

```text
Folder: ...
Markdown files found: ...
Markdown file paths listed: ...
Markdown files with content attached: ...
RAG search recommended: yes ...
Preferred search tool: Hermes MCP/RAG document search ...
Search scope filter: ...
```

이 설계의 의도:

- 플러그인이 모든 파일 내용을 무리하게 프롬프트에 넣지 않음
- `vault-all`에서는 임의 파일 본문 미리보기도 붙이지 않음
- 그래도 Hermes가 큰 범위 질문임을 알 수 있음
- Hermes가 연결된 MCP 검색 도구를 쓰도록 강하게 유도
- 단순 폴더 구조 질문은 `folderFilePaths`만으로 답할 수 있게 함

### 8.2. 테스트된 MCP 이름

현재 테스트 환경에서 Hermes는 MCP 서버를 다음 이름으로 인식했습니다.

```text
obsidian-docsearch
```

확인 명령:

```bash
hermes mcp list
hermes mcp test obsidian-docsearch
```

테스트 결과:

```text
✓ Connected
✓ Tools discovered: 3
doc-ingest
doc-ingest-status
doc-search
```

Hermes 모델이 실제로 본 도구 이름:

```text
mcp_obsidian_docsearch_doc_search
mcp_obsidian_docsearch_list_resources
mcp_obsidian_docsearch_read_resource
```

따라서 코드나 프롬프트에서 `docsearch-mcp`만 쓰면 실제 Hermes 설정명과 어긋날 수 있습니다. 현재 문구는 둘을 함께 언급합니다.

```text
obsidian-docsearch/docsearch-mcp
```

### 8.3. 실제 회귀 테스트 결과

`gpt-5.4-mini`를 사용해 RAG-aware 프롬프트를 Hermes CLI로 테스트했습니다.

테스트 의도:

```text
metadata.obsidian.ragRecommended = true
working scope = vault-all
obsidian-docsearch/docsearch-mcp를 선호 검색 도구로 지시
```

결과적으로 Hermes는 MCP 검색 도구를 사용했고, 다음 문서를 찾아 답했습니다.

```text
hermes continue(obsidian plugin)/docsearch-mcp-hermes-setup-guide.md
2026_오토카/오토카 구조.md
2026_오토카/오토카 프로젝트.md
```

주의:

- Obsidian 플러그인의 dashboard session API 경로는 테스트 시점에 `/api/sessions`, `/api/model/options`, `/api/auth/ws-ticket`가 401을 반환해 직접 라이브 세션 테스트가 막혔습니다.
- 대신 Hermes CLI + 실제 MCP tool discovery + 모델 응답으로 RAG 힌트가 도구 사용을 유도하는지 검증했습니다.
- dashboard 인증 정책이 바뀌면 플러그인 Test Connection부터 다시 확인해야 합니다.

## 9. 파일 경로 정책

### 9.1. Vault 내부 경로

Vault 내부 파일은 기본적으로 vault-relative path를 사용합니다.

예:

```text
컴퓨터/언어/Assembly/예제/이항계수.md
```

장점:

- iCloud, NAS, Windows, macOS에서 Vault 루트가 달라도 의미 유지
- GitHub에 절대경로 노출 방지
- Obsidian API와 잘 맞음

### 9.2. 절대경로가 필요한 경우

Hermes 서버가 직접 filesystem tool로 읽어야 하는 경우 절대경로가 필요할 수 있습니다.

이때 플러그인은 metadata에 vault root hint를 함께 전달합니다.

예:

```text
vaultRoot: /Users/user/Library/Mobile Documents/iCloud~md~obsidian/Documents/컴퓨터
activeFile: 프로젝트/회의록.md
```

Hermes는 필요시 두 값을 결합합니다.

```text
vaultRoot + "/" + activeFile
```

주의:

모바일 Obsidian에서는 서버의 실제 파일 경로를 알 수 없습니다. iPad의 Vault 경로와 Mac Studio의 Vault 경로는 다릅니다. 따라서 플러그인에서 무조건 절대경로를 만들어 보내는 방식은 장기적으로 위험합니다.

### 9.3. Desktop / 바탕화면

사용자가 `바탕화면` 또는 `Desktop`이라고 말하면 이는 Obsidian Vault가 아니라 Hermes 서버 OS의 Desktop입니다.

현재 시스템 지시문은 다음 정책을 전달합니다.

```text
Desktop 또는 바탕화면은 process cwd가 아니라 Hermes server OS의 Desktop directory로 해석한다.
macOS/Linux는 보통 $HOME/Desktop.
Windows는 사용자 Desktop known folder.
애매하면 서버 도구로 확인한 뒤 쓴다.
```

이 정책은 특정 OS에 하드코딩하지 않기 위한 절충입니다.

## 10. Windows 이동 시 경로 문제

Windows에서는 다음 차이가 있습니다.

```text
macOS/Linux: /
Windows: \ 또는 /
Desktop: %USERPROFILE%\Desktop
드라이브: C:\Users\...
```

권장 유지보수 원칙:

1. Vault 내부 파일은 항상 vault-relative path로 유지
2. Obsidian API에는 `/` 기반 path 사용
3. Hermes 서버 도구 실행에는 서버 OS path를 사용
4. 플러그인에서 Windows 절대경로를 임의 조립하지 않기
5. 서버가 실제 파일 작업을 하기 전 `pwd`, home directory, Desktop known folder를 확인하도록 유도

문제가 생기는 대표 사례:

```text
사용자: 바탕화면에 파일 만들어줘
모델: 현재 작업 폴더에 파일 생성
```

대응:

- system message에서 Desktop 해석 규칙 강화
- Hermes tool layer에서 Desktop path helper 제공
- 가능하면 사용자가 명시한 위치를 서버에서 확인한 뒤 write 수행

## 11. iCloud/NAS 동기화 고려

플러그인은 Vault 안에 설치되므로 iCloud/NAS 동기화로 다른 기기에 전달될 수 있습니다.

장점:

- iPad에도 플러그인 파일과 설정이 동기화됨
- 별도 설치 과정이 줄어듦

주의:

- `data.json`도 동기화됩니다.
- username/password, gateway token이 같이 넘어갈 수 있습니다.
- 여러 기기에서 동시에 설정을 바꾸면 충돌 파일이 생길 수 있습니다.
- NAS 경로는 macOS/Windows에서 다르게 mount될 수 있습니다.

권장:

- 개인 단독 사용이면 설정 동기화 허용 가능
- 공유 Vault라면 password 저장 방식을 다시 설계
- 공개 GitHub에는 `data.json` 제외

## 12. Obsidian 업데이트 시 점검할 것

Obsidian 업데이트 후 점검:

- `requestUrl` 동작 여부
- 모바일에서 WebSocket 연결 가능 여부
- `workspace.activeEditor.file` 접근 가능 여부
- `file-open`, `active-leaf-change` 이벤트 동작 여부
- `MarkdownView`, `TFile` type check 동작 여부
- `data.json` 설정 저장/동기화 동작 여부
- CSS 변수명 변경 여부

특히 모바일에서는 브라우저/Electron 차이가 있으므로 desktop에서만 되는 API를 쓰지 않도록 주의해야 합니다.

## 13. Hermes 업데이트 시 점검할 것

Hermes 업데이트 후 점검:

### 인증 API

확인:

```text
/api/auth/providers
/auth/password-login
/api/auth/ws-ticket
```

바뀌면 dashboard login flow 수정이 필요합니다.

### WebSocket API

확인:

```text
/api/ws?ticket=...
session.create
session.resume
prompt.submit
approval.respond
```

RPC method 이름이나 params가 바뀌면 live gateway 연결이 깨집니다.

### 이벤트 이름

확인:

```text
message.delta
message.complete
reasoning.delta
reasoning.available
tool.*
approval.request
```

이벤트 이름이 바뀌면 답변/thinking/tool/approval 표시가 깨질 수 있습니다.

### 모델 API

확인:

```text
/api/model/options
```

provider/model 구조가 바뀌면 모델 드롭다운이 실제 Hermes 앱과 달라질 수 있습니다.

### 세션 API

확인:

```text
/api/sessions
/api/sessions/{sessionId}/messages
```

rename/delete/list 동기화가 깨지면 session parser와 filter를 확인합니다.

## 14. 승인/거절 처리

Hermes가 위험하거나 변경이 필요한 작업을 하려면 approval event를 보낼 수 있습니다.

플러그인 처리 흐름:

```text
Hermes -> approval.request event
Plugin -> approval card render
User -> Approve / Reject click
Plugin -> approval.respond RPC
Hermes -> continue or cancel tool execution
```

유지보수 포인트:

- approval id가 session id와 어떻게 연결되는지 확인
- gateway approval과 REST approval fallback 구분
- 버튼이 안 뜨면 event mapping을 먼저 확인
- 버튼은 뜨는데 작업이 안 이어지면 `approval.respond` params 확인

## 15. 파일 추가/수정/삭제 요청 처리 흐름

Obsidian 플러그인은 파일을 직접 만들거나 삭제하지 않습니다.  
사용자가 Obsidian 채팅창에서 파일 추가, 수정, 삭제를 요청하면 플러그인은 요청과 Obsidian 문맥을 Hermes에 전달하고, 실제 파일 작업은 Hermes 서버가 수행합니다.

기본 흐름:

```mermaid
flowchart TD
  A["User asks in Obsidian<br/>예: 바탕화면에 test.md 만들어줘"] --> B["Plugin builds message"]
  B --> C["Plugin attaches Obsidian context<br/>activeFile, activeFolder, folderFilePaths, rag hints"]
  C --> D["Plugin sends prompt.submit to Hermes live gateway"]
  D --> E["Hermes model interprets request"]
  E --> F["Hermes chooses file/terminal/tool action"]
  F --> G{"Access mode / Hermes policy<br/>approval needed?"}
  G -->|No| H["Hermes executes file operation"]
  G -->|Yes| I["Hermes emits approval.request"]
  I --> J["Plugin renders Approve / Reject buttons"]
  J -->|Approve| K["Plugin sends approval.respond"]
  J -->|Reject| L["Plugin sends deny"]
  K --> H
  H --> M["Hermes streams tool logs and final answer"]
  L --> N["Hermes cancels or changes plan"]
```

### 15.1. 파일 생성 예시

사용자 요청:

```text
바탕화면에 test.md 파일 하나 만들고 "hello"라고 적어줘
```

플러그인이 하는 일:

```text
1. 사용자의 메시지를 Hermes로 전송
2. 현재 Obsidian active file/folder metadata를 함께 전송
3. system message에 Desktop/바탕화면 경로 정책을 포함
4. access mode를 함께 전달
```

Hermes가 하는 일:

```text
1. "바탕화면"을 Hermes 서버 OS의 Desktop directory로 해석
2. 필요하면 서버 도구로 home/Desktop 경로 확인
3. 파일 쓰기 도구 또는 shell 도구를 선택
4. 정책상 승인이 필요하면 approval.request 발생
5. 승인 후 실제 파일 생성
```

중요:

- 생성되는 위치는 iPad의 바탕화면이 아닙니다.
- 생성되는 위치는 Hermes 서버 컴퓨터의 바탕화면입니다.
- Tailscale은 네트워크 연결만 제공하며, 파일 작업은 항상 Hermes 서버에서 일어납니다.

### 15.2. 파일 삭제 예시

사용자 요청:

```text
서버 바탕화면의 test.md 삭제해줘
```

권한 모드가 `Ask before changes`이면 일반적으로 다음 흐름이 기대됩니다.

```text
1. Plugin -> Hermes: 삭제 요청 전송
2. Hermes -> Plugin: approval.request 이벤트 전송
3. Plugin UI: Approve / Reject 버튼 표시
4. User: Approve 클릭
5. Plugin -> Hermes: approval.respond { choice: "once" }
6. Hermes: 실제 삭제 도구 실행
7. Hermes -> Plugin: tool log + 최종 답변 스트리밍
```

삭제 버튼이 뜨지 않는다면 확인할 것:

```text
Use Hermes live gateway = on
Workspace access = Ask before changes 또는 Full access
Hermes가 실제로 approval.request 이벤트를 보내는지
gatewayToLine()에서 approval.request mapping이 유지되는지
approval.respond RPC params가 Hermes 버전과 맞는지
```

현재 폴더 기반 삭제 요청:

```text
현재 폴더에 좀 오래된 글은 좀 삭제해줘
```

이 요청에는 `@현재폴더` mention이 없어도 자연어 감지 로직이 작동합니다.

플러그인이 Hermes에 전달하는 의도:

```text
scope: current-folder
scopePath: 현재 active note의 parent folder
folderFilePaths: 현재 폴더의 Markdown 파일 목록
ragRecommended: true
accessMode: 현재 composer의 access mode
```

그 다음 판단은 Hermes가 합니다.

```text
1. 오래된 글의 기준이 모호하면 Hermes가 기준을 물어볼 수 있음
2. 파일 수정일/내용/이름을 확인하기 위해 파일 또는 terminal tool을 사용할 수 있음
3. 실제 삭제가 필요하면 approval.request를 보낼 수 있음
4. 사용자가 승인하면 삭제 실행
```

중요한 설계 원칙:

- 플러그인은 “현재 폴더”가 어디인지 알려줍니다.
- 어떤 파일이 오래된 글인지 최종 판단하는 것은 Hermes입니다.
- 실제 삭제도 Hermes 서버가 수행합니다.
- 플러그인은 승인 UI와 이벤트 표시만 담당합니다.

### 15.3. Access mode별 기대 동작

플러그인의 access mode는 Hermes에게 전달하는 작업 태도 힌트입니다. 최종 보안 정책은 Hermes 서버가 결정합니다.

```text
Ask before changes
```

- 파일 추가/수정/삭제, shell 실행, install, git 작업 등은 Hermes approval gate를 사용하도록 유도
- 플러그인은 별도 대화형 확인 질문을 만들지 않고 Hermes의 `approval.request`를 UI 버튼으로 표시
- 초보자용 기본값으로 가장 적합

```text
Full access
```

- Hermes 서버 정책이 허용하면 추가 확인 없이 실행할 수 있음
- 위험한 Vault나 공유 서버에서는 신중하게 사용

### 15.4. 경로 해석 원칙

파일 작업 요청에는 두 종류의 경로가 섞일 수 있습니다.

```text
Obsidian Vault 내부 경로
예: 2026_오토카/오토카 구조.md
```

이 경우 플러그인은 Vault 상대 경로를 보냅니다. Hermes는 필요하면 `vaultRoot`와 결합해 서버 파일시스템 경로로 해석합니다.

```text
OS 경로
예: 바탕화면, Desktop, Downloads, /Users/user/Desktop
```

이 경우 Obsidian Vault가 아니라 Hermes 서버 OS 기준으로 해석해야 합니다.

현재 system message 정책:

```text
Desktop 또는 바탕화면은 Hermes server OS desktop directory로 해석한다.
process current working directory나 home folder에 만들지 않는다.
애매하면 서버 도구로 실제 경로를 확인한 뒤 작업한다.
```

Windows 유지보수 포인트:

```text
macOS/Linux: 보통 $HOME/Desktop
Windows: %USERPROFILE%\Desktop 또는 Known Folder API 기준 Desktop
```

플러그인에서 Windows 절대경로를 직접 조립하지 않는 것이 좋습니다.  
Hermes 서버 쪽 tool/helper가 OS별 known folder를 해석하는 방식이 가장 안정적입니다.

### 15.5. 플러그인 코드에서 관련된 부분

파일 작업 요청 자체를 판단하는 핵심은 system message와 gateway event 처리입니다.

관련 키워드:

```text

In Ask/Confirm access mode
contextFromMessage
current-folder
approval.request
approval.respond
gatewayToLine
respondToApproval
Desktop or 바탕화면
vaultRoot
activeFile
folderFilePaths
```

현재 플러그인은 다음 책임까지만 가집니다.

```text

문맥 전달
권한 의도 전달
approval.request 렌더링
Approve/Reject 응답 전송

```

플러그인이 직접 하지 않는 것:

```text

파일 수정
터미널 명령 실행
git commit/push
MCP 도구 직접 호출
```

이 작업들은 Hermes 서버와 Hermes에 연결된 도구가 담당합니다.

## 18. 장기 개선 방향

### TypeScript 소스화

현재 `main.js`가 번들 결과물 중심이라 유지보수가 어렵습니다.

추천 구조:

```text
src/
  main.ts
  hermes-client.ts
  gateway-rpc.ts
  context-router.ts
  mention-menu.ts
  settings-tab.ts
  ui/
styles.css
manifest.json
```

### 공식 API 우선

현재 일부 정보는 Hermes 앱 local storage를 읽어 맞추는 방식이 포함되어 있습니다.

장기적으로는 Hermes가 다음 API를 제공하는 것이 가장 좋습니다.

```text
GET /api/model/visible
PUT /api/model/visible
GET /api/sessions
GET /api/session/{id}
```

공식 API가 생기면 local storage 추정 로직은 제거하는 것이 좋습니다.

### RAG-aware Context Router

큰 Vault를 다루려면 inline context만으로는 한계가 있습니다.

현재 구현 방향:

```text
작은 문맥 -> inline context
큰 문맥 -> metadata + ragRecommended + RAG/MCP search hint
```

Hermes가 MCP 검색 도구를 이미 연결하고 있다면, 플러그인은 사용자 의도를 metadata로 더 정확히 전달하고 Hermes가 검색 도구를 쓰도록 유도합니다.

현재 구현된 큰 문맥 범위:

```text
current-folder
vault-folder
vault-all
```

유지보수할 때 확인할 코드 키워드:

```text
ragRecommended
ragSearchProvider
ragSearchScopeType
ragSearchScopePath
[Folder/Vault structure and RAG hint]
RAG-aware context rule
```

### 플랫폼별 경로 helper

Desktop, Downloads, Documents 같은 OS known folder는 모델 프롬프트에만 맡기지 말고 Hermes tool layer에서 helper로 제공하는 것이 가장 안정적입니다.

예:

```text
resolve_known_folder("desktop")
resolve_known_folder("downloads")
```

이런 helper가 있으면 Windows/macOS/Linux 차이를 모델이 직접 추론하지 않아도 됩니다.
